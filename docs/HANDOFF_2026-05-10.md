## 7. Что **нельзя** трогать без явного запроса пользователя

- **`docker compose`** в `infrastructure/` (особенно блок `n8n.environment`). Нюанс: `WEBHOOK_URL` подхватывается контейнером только при старте, и любой recreate тригерит регистрацию webhook у Telegram. Если что-то идёт не так с webhook — фиксить через `setWebhook` напрямую в Telegram API, не через recreate n8n.
- **Cloudflare named tunnel `gelendzhik-owner-n8n`**, его credentials JSON и DNS-запись CNAME `owner.aquatoring.ru`. Зона `aquatoring.ru` делегирована Cloudflare через NS `katelyn.ns.cloudflare.com`/`ridge.ns.cloudflare.com`. MX/TXT и корневые записи остались импортированными из Timeweb — не править.
- **Telegram webhooks обоих ботов**. Если нужно перерегистрировать — через `deactivate→activate` workflow или через `setWebhook` API; никогда не дёргать с пустым URL «на всякий случай».
- **Polza credential `polza_header_auth`** (зашифрован токен). Менять только через UI n8n или новым credential с переименованием id в БД (как делали с остальными).
- **Активные workflow 01 и 09** — они стабильны, их трогать без причины не нужно.
- **Schema БД `bookings`/`apartments`** — любые ALTER только через явный запрос с backup.
- **Существующие данные `bookings`** — UPDATE/DELETE только через явный запрос. RC-брони синхронизируются автоматически.
- **`infrastructure/cf_watchdog.sh.disabled`** — НЕ переименовывать обратно. Quick-tunnel больше не используется, watchdog только испортит named tunnel.
- **iCal URL в `apartments.ical_url`** — рабочие. Старые backup'ы есть на случай отката, но трогать без необходимости не надо.

## 8. Следующий крупный этап — `message_templates` table

**Зачем**: сейчас все тексты гостевых и owner-сообщений (10 шаблонов в Phase A + B1..B5) живут в JS-коде ноды `Select Message` в workflow 03. Чтобы поправить формулировку, надо открывать JSON и реимпортировать workflow. Хочется иметь возможность менять тексты без кода.

**Идея**: вынести шаблоны в Postgres-таблицу с ключом-событием.

Минимальный план (не делать без явного «go» от пользователя):

1. Backup workflow 03 → `workflows/backups/03_guest_journey_before_message_templates_<TS>.json`.
2. SQL-миграция:
   ```sql
   CREATE TABLE IF NOT EXISTS message_templates (
     key       VARCHAR(80) PRIMARY KEY,         -- например 'guest_d_minus_1_checkin'
     channel   VARCHAR(16) NOT NULL,            -- 'guest' | 'owner' | 'owner_brief'
     lang      VARCHAR(8)  NOT NULL DEFAULT 'ru',
     body      TEXT        NOT NULL,            -- текст с плейсхолдерами {{apartment_id}} и т.п.
     active    BOOLEAN     NOT NULL DEFAULT TRUE,
     updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
   );
   ```
3. Seed данных: текущие 10 шаблонов из `Select Message` JS-кода (5 guest_* + 4 owner_* + owner_brief). Можно через INSERT-SQL.
4. Workflow 03 → новая Postgres-нода `Load Templates` (executeOnce) перед `Select Message`. `Select Message` берёт `body` из загруженных шаблонов и подставляет плейсхолдеры через простой `replaceAll(/\{\{(\w+)\}\}/g, ...)`. Если в `message_templates` нужного ключа нет — fallback на JS-шаблон, чтобы старая логика не сломалась внезапно.
5. Manual test (через `manual-journey-bX-test-*` бронь, как делалось в Phase B1..B5).
6. Reactivate, idempotency check, cleanup тестовой брони.
7. Обновить release-doc — раздел 5.7.

**Что не делать**:
- Не пытаться сразу делать UI/admin-panel поверх таблицы. Сначала база.
- Не тащить в `message_templates` тексты Owner Bot ответов — они генерируются Polza, шаблонами там не управляют.
- Не менять регулярки `Build Context` под этим этапом — парсер дат живёт отдельно.

После этого следующий по приоритету шаг — **Vercel admin panel** (CRUD для `message_templates` + просмотр `bookings.journey_sent`), но это уже отдельная инфраструктурная задача, нужен решённый фронт-контур.

```bash
# Состояние workflow и executions
docker exec gelendzhik-postgres psql -U n8n_user -d n8n_gelendzhik -c \
  "SELECT id, name, active, \"updatedAt\" FROM workflow_entity WHERE active=true ORDER BY name;"

docker exec gelendzhik-postgres psql -U n8n_user -d n8n_gelendzhik -c \
  "SELECT id, \"workflowId\", status, mode, \
   EXTRACT(EPOCH FROM (\"stoppedAt\"-\"startedAt\"))::int AS sec, \"startedAt\" \
   FROM execution_entity ORDER BY id DESC LIMIT 10;"

# Telegram webhook health
OWNER_TOK=$(grep "^TELEGRAM_OWNER_BOT_TOKEN=" \
  /Users/Orlova/gelendzhik-guesthouse-automation/infrastructure/.env | cut -d= -f2)

# Доступность named tunnel
curl -s -o /dev/null -w "HTTP %{http_code}\n" https://owner.aquatoring.ru/rest/login

# Декодирование execution_data (артефакт ссылочного формата n8n)
docker exec gelendzhik-postgres psql -U n8n_user -d n8n_gelendzhik -tA -c \
  "SELECT data FROM execution_data WHERE \"executionId\"=<ID>;" > /tmp/x.json
# затем python с рекурсивным deref как в моих предыдущих сообщениях

# Manual run workflow (использовался для всех тестов Phase B)
WF_ID=<workflow_id>
curl -s -c /tmp/c.txt -X POST http://localhost:5678/rest/login \
  -H 'Content-Type: application/json' \
  -d '{"emailOrLdapLoginId":"qb.taxation@gmail.com","password":"Gelendzhik2026!"}'
WF=$(curl -s -b /tmp/c.txt "http://localhost:5678/rest/workflows/$WF_ID")
PAYLOAD=$(jq -c '{workflowData:{id:.data.id,name:.data.name,nodes:.data.nodes,
  connections:.data.connections,settings:(.data.settings // {executionOrder:"v1"}),
  pinData:(.data.pinData // {}),active:false,versionId:.data.versionId},
  runData:{},triggerToStartFrom:{name:"<NAME>",data:{json:{}}}}' <<< "$WF")
curl -s -b /tmp/c.txt -X POST "http://localhost:5678/rest/workflows/$WF_ID/run" \
  -H 'Content-Type: application/json' -d "$PAYLOAD"
```
